Administrable and Activatable Service

Let's try to create the service that is activatable and supports remote adminstration.

This example is based on remote primer. We will modify remote filter implementation as administrable (org.sf.jini.examples.adm.AdmFilterImpl) with activatable support (org.sf.jini.examples.admact.AdmActFilterImpl). The last class is implemented as activatable object.

The service (org.sf.jini.examples.admact.AdmFilterService) registers corresponding proxy object, whereas the client (org.sf.jini.examples.adm.AdmFilterClient) is looking for required service.

After the buildt we have 3 archives:
  • for service itself (target/lib/adm-filter-service.jar);
  • for client that wants to work with this service (target/docroot/service/adm-filter-service-dl.jar);
  • for activation framework (target/docroot/service/adm-filter-service-act.jar).


    • Now it's time to test them. Execute these commands in the following order:

    >start ant start.services
    >start ant phoenix.start
    >ant admact.service
    >ant admact.client

    The client will check the interfaces, implemented by proxy object and if it implements Administrable interface, the client will try to remove the service.

    To check all these events in dynamic manner, look at jini browser window. You will see how service is added to registrar and how it will be deleted. Don't forget to select the registrar from the Regisrar menu item.

    In this example the service also uses more sofisticated way to inform the client part about it's details on the bas of the ServiceInfo class instead of the Name.
